# V1 限时挑战技术实现文档

## 1. 玩法定位

限时挑战模式的目标是在固定时间内完成尽可能多的正确分类，并在结算时展示得分、正确率、处理数量和错误热点。

该模式与当前主干最大的区别不是分类逻辑变化，而是：

- 不再按固定目标数量结束
- 场景垃圾需要动态补充
- 成绩重心从“是否完成目标”转向“时间内拿到多少分”

## 2. 核心规则

建议首版规则：

- 开局时间固定，例如 `120` 秒
- 场景始终维持 `N` 个活跃垃圾，例如 `4-6` 个
- 玩家每完成一次投放后，系统补充一个新的随机垃圾
- 正确投放加分，错误投放扣分或少量扣分
- 时间结束立即结算

推荐首版得分：

- 正确：`+100`
- 错误：`-25`
- 不做连击
- 不做额外时间奖励

这样可以保证首版规则直观、好调试、好展示。

## 3. 与主干复用关系

必须复用：

- `GarbageItem`
- `DropZone.TryClassifyReleasedItem`
- `ClassificationEvents`
- `DesktopGarbageInteractor`
- `WasteAnalyticsTracker`
- `WasteHudView`
- `WasteResultView`

限时挑战不应重写垃圾投放判定，只在“模式控制”和“垃圾生成调度”上扩展。

## 4. 技术实现方案

### 4.1 模式控制器

建议新增：

```text
Assets/Scripts/Modes/TimedChallenge/TimedChallengeModeController.cs
Assets/Scripts/Modes/TimedChallenge/TimedChallengeConfig.cs
```

职责：

- 初始化时间、分数和活跃垃圾数量
- 开局时生成初始垃圾
- 接收分类结果并结算分数
- 处理垃圾补充
- 时间结束后输出结果摘要

### 4.2 垃圾生成器

建议新增：

```text
Assets/Scripts/Modes/TimedChallenge/TimedChallengeSpawner.cs
Assets/Scripts/Modes/TimedChallenge/TimedChallengeSpawnPoint.cs
```

实现要点：

- 场景内预放若干 `SpawnPoint`
- 开局随机挑选若干点位生成垃圾
- 每处理完一个垃圾，立即补一个新垃圾到空闲点位
- 新垃圾来源于配置中的垃圾池

为了避免复杂对象池拖慢开发，首版可以直接实例化和销毁，但文档中要保留后续改为对象池的空间。

### 4.3 活跃垃圾数量控制

建议模式层维护：

- 当前活跃垃圾列表
- 当前空闲出生点列表
- 已处理总数

逻辑：

1. 开局填满目标活跃数
2. 正确或错误处理后，把该垃圾从活跃列表移除
3. 尝试生成新垃圾补足活跃数

错误投放后，如果仍保留原物体重试，则不要立即补垃圾；如果错误后物体直接结束并替换，则应立刻补垃圾。

首版建议：

- 错误投放也视为本次机会已结算
- 当前垃圾结束并刷出新垃圾

这样更符合“挑战模式”的节奏。

## 5. 配置方案

建议配置类包含：

- `timeLimitSeconds`
- `activeGarbageCount`
- `availableGarbageItemIds`
- `scorePerCorrect`
- `penaltyPerWrong`
- `spawnPointGroupId`

建议将可选垃圾池做成：

- 直接引用 `WasteContentCatalog`
- 再额外配置“允许出现的 itemId 白名单”

这样可以复用已有资源定义。

## 6. UI 与结算

### 6.1 HUD 扩展

该模式 HUD 至少需要显示：

- 剩余时间
- 当前得分
- 已处理数量
- 当前反馈

可选扩展：

- 正确率
- 每分钟处理效率

首版建议先不在 HUD 上堆太多指标，避免影响当前可读性。

### 6.2 结算扩展

结果页建议新增或重点展示：

- 模式名：限时挑战
- 最终得分
- 正确数 / 错误数
- 正确率
- 总处理数
- 错误最多的垃圾
- 错误类别分布说明

影响文案建议偏挑战导向，例如：

> 你在限定时间内完成了 18 次分类，其中 14 次正确。最容易混淆的是奶茶杯，建议下一轮重点练习其他垃圾判断。

## 7. 数据统计建议

建议在现有 `ClassificationRecord` 基础上，额外在 summary 层统计：

- `TotalProcessedCount`
- `MostMistakenItemName`
- `MostMistakenItemCount`
- `MistakeCountByCategoryPair`

统计方式可以在模式结束时由控制器遍历 `WasteAnalyticsTracker.Records` 计算，不要求一开始就大改 tracker 内部结构。

## 9. 分支与改动边界

推荐分支：

```text
codex/feature-v1-timed-challenge
```

主要改动区域：

- `Assets/Scripts/Modes/TimedChallenge/`
- 少量 `Assets/Scripts/Core/`
- 少量 `Assets/Scripts/UI/`
- 少量 `Assets/Scripts/Analytics/`

不要做的事：

- 不重写 `DesktopGarbageInteractor`
- 不复制一份新的结果页系统
- 不把无尽刷分和闯关规则一起写进本模式控制器

## 10. 联调注意事项

- 如果扩展 `WasteSessionSummary`，必须保证旧主流程还能正常显示结果
- 如果需要新的场景配置入口，应优先通过可选组件挂接，而不是改死现有主干流程
- 动态生成垃圾要确保带上 `GarbageItem`、Collider、Rigidbody 等必要组件

## 11. 验收标准

1. 可以进入限时挑战模式。
2. 开局能生成指定数量垃圾。
3. 正确或错误处理后能补新垃圾。
4. 时间结束能正常结算。
5. 结算页能看到得分、正确率、总处理数和错误热点。
6. 不影响现有基础模式编译和运行。
