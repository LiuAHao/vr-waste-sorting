# V1 标准闯关技术实现文档

## 1. 玩法定位

标准闯关模式是 V1 最重要的主线玩法，也是最适合作为默认演示入口的一条线。

该模式的目标不是单纯刷分，而是通过逐步增加垃圾种类和时间限制，形成清晰的难度曲线，让玩家从基础分类逐步过渡到更复杂的真实生活垃圾判断。

这是 V1 最接近“教学产品主流程”的模式。

## 2. 核心规则

建议首版按关卡配置推进，每关定义：

- 目标数量
- 时间限制
- 可出现垃圾种类池
- 关卡名称

推荐首版三关结构：

1. 第 1 关：基础垃圾少量种类，时间宽松
2. 第 2 关：加入更多类别，目标数量增加
3. 第 3 关：加入易混淆垃圾，时间更紧

每关完成条件：

- 在时间内完成本关目标数量

失败条件：

- 时间结束仍未完成目标

首版建议不做复杂星级评价，只做：

- 通关 / 未通关
- 得分
- 正确率
- 用时

## 3. 与现有主干的关系

标准闯关是最适合建立在当前主干之上的模式，因为现有 `WasteGameFlowController` 已经具备：

- 倒计时
- 进度
- 得分
- 分类反馈
- 结果结算

因此这条线最合理的做法不是推倒重写，而是把“单轮主流程”扩展成“关卡序列流程”。

## 4. 技术实现方案

### 4.1 关卡配置

建议新增：

```text
Assets/Scripts/Modes/StageProgression/StageProgressionConfig.cs
Assets/Scripts/Modes/StageProgression/StageDefinition.cs
```

每个 `StageDefinition` 建议包含：

- `stageId`
- `stageName`
- `targetCount`
- `timeLimitSeconds`
- `availableGarbageItemIds`
- `initialSpawnCount`

### 4.2 闯关控制器

建议新增：

```text
Assets/Scripts/Modes/StageProgression/StageProgressionModeController.cs
```

职责：

- 管理当前关索引
- 读取当前关配置
- 初始化本关垃圾池与目标
- 监听本关完成或失败
- 决定进入下一关或结束整轮闯关

### 4.3 本关垃圾生成

建议新增：

```text
Assets/Scripts/Modes/StageProgression/StageGarbageSpawner.cs
```

逻辑建议：

- 每关开始时生成固定数量垃圾
- 本关内不需要像限时挑战那样无限补货
- 允许“目标数量 <= 生成数量”，但首版建议两者保持一致，减少理解成本

为了与限时挑战、无尽刷分区分，标准闯关更适合“每关固定题量”的节奏。

## 5. 难度曲线建议

用户已经确认该模式采用：

- 标准闯关优先
- 逐步增加种类
- 逐步收紧时间限制

因此推荐难度变化只围绕两件事：

1. 垃圾池变大
2. 时间变短

首版不要同时再加：

- 连击要求
- 错误次数上限
- 随机关卡事件

否则会让“闯关主线”变得不够清晰。

## 6. UI 与结算

### 6.1 HUD

需要在现有 HUD 基础上补充：

- 当前关卡名或关卡序号
- 当前关进度
- 剩余时间
- 当前得分

例如：

- `第 1 关：基础分类`
- `第 2 关：混合场景`
- `第 3 关：易混淆挑战`

### 6.2 关间反馈

建议在每关结束后增加一个轻量关间面板：

- 本关完成
- 本关正确率
- 下一关目标预告

首版如果不想加中间面板，也可以：

- 本关结束后直接切下一关
- 通过 HUD 短提示显示“进入下一关”

但文档上推荐保留关间过渡的设计空间。

### 6.3 最终结算

最终结果页建议展示：

- 模式名：标准闯关
- 通关关数
- 总正确率
- 总得分
- 总用时
- 错误最多垃圾
- 卡关关卡

如果全部通关，可展示：

- “完成全部关卡”

如果中途失败，可展示：

- “止步第 X 关”

## 7. 数据与展示扩展

该模式适合增加：

- `ClearedStageCount`
- `CurrentStageIndex`
- `FailedStageIndex`
- `MistakesByStage`

答辩时可以展示：

- 玩家在不同难度关卡中的表现差异
- 哪一关开始出现明显错误上升
- 哪类垃圾在高难关卡中最易混淆

## 9. 分支与改动边界

推荐分支：

```text
codex/feature-v1-stage-progression
```

主要改动区域：

- `Assets/Scripts/Modes/StageProgression/`
- 少量 `Assets/Scripts/Core/`
- 少量 `Assets/Scripts/UI/`
- 少量 `Assets/Scripts/Analytics/`

注意：

- 不要把无限补货逻辑混入闯关控制器
- 不要把限时挑战的成绩规则直接套进闯关模式

## 10. 联调注意事项

- 每关切换时要正确清理上一关活跃垃圾
- 关卡切换不能导致 UI 重复创建或旧状态残留
- 如果使用统一结果页，要保证既支持“单轮失败”，也支持“全部通关”

## 11. 验收标准

1. 可以进入标准闯关模式。
2. 至少能配置并运行 3 个关卡。
3. 后续关卡的垃圾池和时间限制会提高难度。
4. 当前关完成后能顺利进入下一关。
5. 最终结算能显示通关情况和错误热点。
6. 模式可作为默认演示入口，不破坏现有基础主流程。
