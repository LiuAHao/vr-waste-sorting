# ParkClean VR V0.3 MVP 初步规划设计

## 1. V0.3 版本定位

V0.3 的目标是先跑通一个“键盘鼠标可玩的 Unity 垃圾分类 MVP”，不把 VR 适配作为本阶段主要任务。

当前项目原型来自“清扫社区”Demo，已有场景、垃圾素材、点击清理、倒计时和胜负面板。V0.3 要做的不是重写一个完整 VR 系统，而是在现有基础上把核心玩法从“清扫垃圾”升级为“垃圾分类”：

> 玩家用键盘鼠标在场景中移动和观察，用鼠标选中/抓取垃圾，将垃圾拖放或投放到四分类垃圾桶，系统判断正误，给出提示，最终展示正确率和错误复盘。

本阶段优先保证能玩、能演示、能解释设计价值。所有复杂功能都先收住，避免开发范围失控。

## 2. V0.3 不做什么

为了让开发团队能在有限时间内稳定交付，V0.3 明确不做以下内容：

| 暂不做 | 原因 |
| --- | --- |
| VR 头显和手柄适配 | 已有人负责后续适配，本阶段只保留可迁移的交互接口 |
| 手势识别 | 成本高，且不是键鼠 MVP 必需 |
| 多场景切换 | 先完成一个校园食堂或社区场景 |
| 复杂投掷物理 | 容易调试失控，先做稳定抓取/拖放/判定 |
| 连击、成就、排行榜 | 会增加 UI 和状态管理复杂度 |
| 复杂环保影响模拟 | V0.3 只在结算页用文案表达 |
| 联网数据上传 | 本地统计即可 |
| 完整资产重构 | 只整理 MVP 所需资产，不做大规模目录迁移 |

## 3. V0.3 成功标准

V0.3 完成时，需要满足以下标准：

1. 玩家能从游戏开始进入一个可操作场景。
2. 场景中有四类垃圾桶：可回收物、有害垃圾、厨余垃圾、其他垃圾。
3. 场景中有 8-12 个可分类垃圾物品。
4. 每个垃圾物品都有名称、正确分类和错误解释。
5. 玩家可以用键盘鼠标完成选中、抓取、移动、投放。
6. 垃圾进入垃圾桶区域后，系统能判断分类是否正确。
7. 正确投放会计入进度，错误投放会提示原因并允许重试。
8. 倒计时、进度、分数或正确数能正常显示。
9. 完成或超时后出现结算页，展示正确率、用时、错误物品。
10. 代码结构能让后续 VR 适配人员复用分类规则和任务逻辑。

## 4. 推荐团队分工

本阶段建议把三名开发拆成三个相对独立的模块，减少互相踩文件和反复返工。

### 4.1 开发 A：核心分类逻辑与垃圾桶判定

负责范围：

- 垃圾分类枚举。
- 垃圾物品数据组件。
- 垃圾桶组件。
- 投放触发区。
- 分类判断逻辑。
- 正确/错误结果事件。

建议文件：

```text
Assets/Scripts/Gameplay/WasteCategory.cs
Assets/Scripts/Gameplay/GarbageItem.cs
Assets/Scripts/Gameplay/TrashBin.cs
Assets/Scripts/Gameplay/DropZone.cs
Assets/Scripts/Gameplay/ClassificationResult.cs
```

核心设计：

```csharp
public enum WasteCategory
{
    Recyclable,
    Hazardous,
    Kitchen,
    Other
}
```

`GarbageItem` 挂在每个垃圾物品上，至少包含：

- `itemName`：垃圾名称。
- `category`：正确分类。
- `wrongReason`：错误提示文案。
- `startPosition`：初始位置，用于错误复位。
- `isCompleted`：是否已经正确分类。

`TrashBin` 挂在垃圾桶上，至少包含：

- `category`：垃圾桶类别。
- `displayName`：中文名称。
- `highlightRenderer`：可选，用于正确/错误高亮。

`DropZone` 放在垃圾桶桶口或桶身前方的触发区，负责检测垃圾进入区域。它只做一件事：拿到 `GarbageItem` 和 `TrashBin` 后发起分类判断。

开发边界：

- 不负责 UI 长什么样。
- 不负责玩家如何抓取垃圾。
- 不负责结算页排版。
- 只需要暴露“分类正确/分类错误”的结果给其他模块。

验收方式：

- 在场景中手动把垃圾物体拖进不同桶的触发区域，能正确打印或触发正误结果。
- 正确分类后垃圾不会重复计分。
- 错误分类后垃圾仍可再次尝试。

### 4.2 开发 B：键鼠交互、玩家控制与可玩流程

负责范围：

- 保留或简化现有键盘移动、鼠标视角。
- 鼠标选中垃圾。
- 鼠标抓取/拖动垃圾。
- 释放垃圾。
- 与开发 A 的 `GarbageItem`、`DropZone` 对接。
- 保证键鼠版本能完整玩一轮。

建议文件：

```text
Assets/Scripts/Player/PlayerController.cs
Assets/Scripts/Interaction/DesktopGarbageInteractor.cs
Assets/Scripts/Interaction/SelectableHighlighter.cs
```

推荐交互方案：

V0.3 不做复杂抛掷，采用“鼠标射线选中 + 按住拖动 + 松开放下”的稳定方案。

操作方式：

1. WASD 移动。
2. 鼠标控制视角。
3. 鼠标左键点击垃圾后进入抓取状态。
4. 垃圾跟随摄像机前方固定距离或鼠标射线命中点移动。
5. 再次点击或松开左键释放垃圾。
6. 垃圾进入垃圾桶触发区域后由开发 A 的逻辑判断。

抓取实现建议：

- 通过从摄像机发射 Raycast 选中带 `GarbageItem` 的物体。
- 抓取后暂时关闭或弱化物理影响，避免物体乱飞。
- 抓取时让物体保持在摄像机前方 1.5-2.5 米。
- 释放后恢复物理或直接保持当前位置。

开发边界：

- 不负责垃圾分类规则。
- 不负责结算统计。
- 不负责美术摆放。
- 只保证玩家能稳定拿起、移动、放下垃圾。

验收方式：

- 玩家能在不使用 VR 设备的情况下完成一轮垃圾投放。
- 抓取过程不穿模到不可找回位置。
- 垃圾投错后能继续被抓取。
- 垃圾投对后不再被重复抓取或重复计入。

### 4.3 开发 C：任务流程、UI 反馈与结算统计

负责范围：

- 游戏开始、进行中、结束状态。
- 倒计时。
- 当前进度。
- 正确数、错误数、得分。
- 正确/错误提示。
- 错误物品记录。
- 结算页。

建议文件：

```text
Assets/Scripts/Core/GameManager.cs
Assets/Scripts/Core/TaskController.cs
Assets/Scripts/UI/HUDController.cs
Assets/Scripts/UI/FeedbackPanel.cs
Assets/Scripts/UI/ResultPanel.cs
Assets/Scripts/Analytics/SessionStats.cs
```

核心流程：

```text
开始游戏
-> 初始化倒计时、目标数量、统计数据
-> 玩家投放垃圾
-> 接收分类结果
-> 正确：进度 +1，得分 +10，显示正确提示
-> 错误：错误次数 +1，记录错误物品，显示原因
-> 判断是否完成目标或倒计时结束
-> 显示结算页
```

UI 最小要求：

- 游戏中显示：倒计时、进度、分数、当前提示。
- 错误提示显示：物品名称、正确分类、原因。
- 结算页显示：完成/失败、得分、正确率、用时、错误列表、环保影响短文案。

结算页文案示例：

> 本轮你完成了 10 件垃圾分类，正确率 83%。继续练习易混淆物品，可以减少现实生活中的错误投放。

开发边界：

- 不负责垃圾物品抓取。
- 不负责垃圾桶触发判定。
- 不负责大规模 UI 美术设计。
- UI 先能清楚展示信息，后续再美化。

验收方式：

- 正确投放会更新进度。
- 错误投放会显示原因并记录到结算。
- 倒计时结束会进入失败结算。
- 全部完成会进入成功结算。
- 重开游戏能清空上一轮数据。

## 5. 统筹与方案/AI 建模工作

V0.3 采用“资产先行，开发后接入”的节奏。也就是说，先由统筹侧完成本阶段需要的垃圾模型、垃圾桶模型、模型清单和分类数据表，等资产范围基本冻结后，三名开发再进入功能开发。

这样做的好处是：开发不需要先用大量占位物搭系统再反复替换，分类数据、模型名称、Prefab 命名和错误解释也能在一开始统一，后续更容易分工。

### 5.1 统筹需要锁定的内容

1. V0.3 只做一个场景。
2. V0.3 先生成完整 12 个垃圾模型；如果开发时间紧，可以只接入其中 8 个。
3. V0.3 垃圾分类口径使用简化版本。
4. 所有错误解释文案在开发前给定。
5. 四分类垃圾桶模型、颜色、文字、图标先按四分类标准走。
6. AI 模型资产先追求可识别、比例适合交互，不追求高精度。
7. 模型交付后尽量冻结名称、分类和数量，除非发现阻塞性问题。

### 5.2 V0.3 垃圾清单建议

统筹侧先生成 12 个模型，开发侧按“优先接入 8 个、再补齐 12 个”的方式控制风险。

| 分类 | 8 件基础版 | 12 件完整版补充 |
| --- | --- | --- |
| 可回收物 | 塑料瓶、纸箱 | 易拉罐 |
| 厨余垃圾 | 剩饭、果皮 | 菜叶 |
| 有害垃圾 | 旧电池、过期药品 | 灯管 |
| 其他垃圾 | 污染纸巾、奶茶杯 | 油污外卖盒 |

### 5.3 资产交付标准

开发正式接入前，统筹侧需要交付一版“V0.3 资产包”，至少包含：

- 12 个垃圾模型文件，或已导入 Unity 的模型资源。
- 4 个垃圾桶模型文件，或已导入 Unity 的模型资源。
- 每个模型的建议缩放比例和摆放备注。
- 每个垃圾的 ID、中文名、英文分类、错误解释。
- 哪 8 个属于基础接入范围，哪 4 个属于补充范围。
- 如某个模型暂时不够理想，也要标记“可先用”或“需替换”。

模型验收不追求精修，但需要满足：

- 玩家一眼能看出大概是什么物品。
- 易混淆垃圾要有必要特征，例如奶茶杯有吸管或残留感、外卖盒有油污感。
- 模型大小不要离谱，导入 Unity 后能通过统一缩放快速调整。
- Collider 可以先用 BoxCollider、SphereCollider 或 CapsuleCollider 简化处理。

## 6. 模块协作接口

为了减少三人之间互相等待，V0.3 建议先约定几个简单接口。

### 6.1 分类结果事件

开发 A 在分类判断完成后，向外提供结果：

```text
GarbageItem item
TrashBin bin
bool isCorrect
string reason
```

开发 C 只关心这个结果，不需要知道判断过程。

### 6.2 垃圾物品状态

每个垃圾物品至少有三种状态：

| 状态 | 说明 |
| --- | --- |
| Idle | 未被抓取，可交互 |
| Held | 正在被玩家抓取 |
| Completed | 已正确分类，不再交互 |

开发 B 负责从 `Idle` 切到 `Held` 和释放；开发 A 或 C 负责正确分类后切到 `Completed`。

### 6.3 UI 更新接口

开发 C 提供简单方法，供其他模块调用或由事件驱动：

```text
ShowCorrect(itemName)
ShowWrong(itemName, correctCategory, reason)
UpdateProgress(done, total)
ShowResult(stats)
```

## 7. 推荐开发顺序

### 第 0 步：统筹先交付 V0.3 资产包

负责人：统筹。

目标：

- 生成 12 个垃圾模型和 4 个垃圾桶模型。
- 输出固定的数据表：ID、名称、分类、错误解释、模型状态。
- 标记优先接入的 8 个垃圾和补充接入的 4 个垃圾。
- 将模型放到明确目录，例如 `3D-Model/V0.3/` 或导入 Unity 后放到 `Assets/Art/GarbageItems/`。

完成标准：

- 三名开发可以不再等待模型命名和分类口径。
- 资产可以不是最终精修版，但必须能用于接入和演示。

### 第 1 步：搭出键鼠可移动场景

负责人：开发 B。

目标：

- 玩家能进入场景。
- 能用 WASD 和鼠标观察。
- 场景中能看到基础环境、四个桶和第一批垃圾模型。

### 第 2 步：给垃圾和桶加分类属性

负责人：开发 A。

目标：

- 每个已接入垃圾有分类。
- 每个桶有分类。
- 触发区域能判断正误。
- 优先接入 8 个基础垃圾；如果顺利，再补齐 12 个。

### 第 3 步：跑通抓取和投放

负责人：开发 B + 开发 A。

目标：

- 玩家能抓垃圾。
- 垃圾能放到桶附近。
- 系统能收到分类结果。

### 第 4 步：接入 UI 和任务流程

负责人：开发 C。

目标：

- 正确进度、错误提示、倒计时都能显示。
- 完成或超时能结算。

### 第 5 步：补齐内容和演示细节

负责人：统筹 + 三名开发。

目标：

- 如果基础 8 个已稳定，补齐到 12 个垃圾。
- 检查模型缩放、摆放、Collider 是否影响抓取和投放。
- 检查错误解释是否和模型表现一致。

### 第 6 步：整体验收和收口

负责人：三名开发 + 统筹。

目标：

- 从开始到结算完整跑通。
- 不新增功能，只修阻塞问题。

为了减少冲突，建议三名开发分开负责目录：

| 人员 | 主要目录 | 说明 |

## 10. 开发控制原则

V0.3 开发过程中，遇到新想法先按以下规则判断：

1. 不能帮助“键鼠完整跑通一轮分类”的功能，先不做。
2. 需要超过半天调试的视觉效果，先降级。
3. 会影响三个人共同接口的改动，先讨论再写。
4. 新增垃圾数量超过 12 个，先不加。
5. UI 先清楚，再美观。
6. 物理先稳定，再真实。
7. VR 适配只做准备，不提前进入主线。

## 11. V0.3 到 VR 适配的预留点

虽然 V0.3 不做 VR，但代码要避免把键鼠输入和分类逻辑绑死。

需要预留：

- 分类判断只依赖 `GarbageItem` 和 `TrashBin`，不依赖鼠标。
- 任务流程只接收分类结果，不关心结果来自键鼠还是 VR 手柄。
- 抓取逻辑单独放在 `DesktopGarbageInteractor`，以后可替换成 `XRGarbageInteractor`。
- UI 文案和结算统计不要写在玩家控制脚本里。

这样后续 VR 适配时，只需要替换输入和抓取层，分类规则、任务流程、反馈统计都可以复用。
