# ParkClean VR MVP 收口与整体完善计划

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 基于当前 `main` 分支，把已经跑通的垃圾分类桌面端 MVP 收口成一个可稳定演示、可继续维护、可为后续 VR 适配复用的版本。

**Architecture:** 当前项目已经形成 `Gameplay + Interaction + Core/UI` 的三层骨架，问题主要集中在可见层乱码、测试残留、旧逻辑兼容方式过脆、场景与文档编码污染。收口方案不再新增玩法，只围绕“修阻塞项、稳主流程、补验收基线”推进。

**Tech Stack:** Unity、C#、内置 UGUI、场景运行时自动注入、基于事件的分类结果广播

---

## 一、现状摘要

当前 `main` 已经具备以下能力：

- 场景内存在垃圾物体、垃圾桶、投放判定与分类事件
- 桌面端已经支持键鼠移动、视角控制、鼠标抓取和释放垃圾
- 主流程已接入倒计时、计分、正确/错误反馈、结算页和重开
- 正确投放后不会重复计分，错误投放会扣分并允许重试

当前确认存在的主要问题：

1. 用户可见 UI 文案和部分计划文档出现乱码
2. 场景里仍保留 `ClassificationTestListener` 测试监听对象
3. 旧 HUD 的隐藏逻辑依赖文本匹配，且匹配文本本身已乱码
4. Unity Package Manager 在线搜索报 `ECONNRESET`
5. 运行中看到“`[垃圾袋] 投进 [可回收物] | 正确:False`”这类日志，说明测试监听还在工作

## 二、对这次新提示的判断

### 1. Unity Package Manager 联网报错

现象：

- `Unable to perform online search`
- `Cannot connect to 'packages.unity.com'`
- `error code: ECONNRESET`

判断：

- 这不是当前游戏逻辑代码的直接 bug
- 更像当前网络环境、代理、防火墙、校园网或 Unity 包服务连接异常
- 只要当前项目依赖已经在本地解析完成，它不会阻止场景运行和本轮 MVP 收口

处理原则：

- 不把它作为本轮主线修复项
- 先记为“环境问题/可选排查项”
- 只有在后续确实需要新增或升级包时，再单独检查代理、镜像和防火墙策略

### 2. `ClassificationTestListener` 日志输出

现象：

- 投错垃圾后控制台输出：
  - `[垃圾袋] 投进 [可回收物] | 正确:False | 原因:垃圾袋属于有害垃圾。`

判断：

- 这条日志说明分类事件链路是通的
- 也说明场景里测试监听器仍然启用
- 对开发调试有帮助，但正式演示中属于测试残留，不建议继续保留在主场景启用态

处理原则：

- 从正式场景移除或默认禁用 `TestListener`
- 如果团队仍希望保留调试能力，可以改为仅开发场景挂载，或加 `Debug.unityLogger.logEnabled` 风格的可控开关

## 三、收口目标

本轮收口完成后，需要达到以下结果：

1. 主流程所有中文文案正常显示
2. `docs/plan/` 中至少核心计划文档能正常阅读
3. 正式场景不再包含会干扰演示的测试监听和旧 UI 叠层
4. 进入场景后可以稳定完成一轮：
   - 抓取垃圾
   - 正确投放
   - 错误投放并重试
   - 超时结算
   - 全部完成结算
   - 点击重开
5. 当前版本形成一份明确的 MVP 验收清单

## 六、建议验收顺序

在所有 P0/P1 修复完成后，按下面顺序做一次完整验收：

1. 打开 Unity，确认不再出现旧 UI 叠层
2. 忽略 Package Manager 联网搜索异常，先验证场景可运行
3. 进入主场景，观察 HUD 中文是否正常
4. 抓起一个垃圾物体，确认高亮、抓取、跟随正常
5. 正确投放一个垃圾，确认：
   - 加分
   - 进度更新
   - 正确反馈显示正常
6. 错误投放一个垃圾，确认：
   - 扣分
   - 显示错误原因
   - 物体回到起点
   - 可再次抓取
7. 完成全部目标，确认成功结算
8. 重新开局，确认统计归零
9. 额外跑一轮超时流程，确认失败结算

## 七、执行建议

如果只看“能不能进入下一轮 PR/演示”，建议优先级如下：

1. 先修代码乱码
2. 再修计划文档乱码
3. 再移除 `TestListener`
4. 再改旧 UI 清理逻辑
5. 最后补验收清单

这样做的好处是：

- 最快恢复用户可见质量
- 最快降低误判和调试噪音
- 不会把时间花在暂时不阻塞的网络环境问题上
