# V0.3 开发协作规范

## 1. 基本原则

V0.3 的目标是先跑通一个键盘鼠标可玩的 Unity 垃圾分类 MVP。三名开发需要分模块并行，但不能直接在主分支上改，也不能开发完成后直接合入。

所有开发完成后必须提交 PR，由统筹负责人进行代码审核，确认没有问题后再合入。

## 2. 分支规范

每名开发都需要从主分支 checkout 自己的功能分支。

建议分支名：

| 开发 | 负责模块 | 分支名 |
| --- | --- | --- |
| 开发 A | 核心分类逻辑与垃圾桶判定 | `codex/feature-classification-core` |
| 开发 B | 键鼠交互与玩家控制 | `codex/feature-desktop-interaction` |
| 开发 C | 任务流程、UI 反馈与结算统计 | `codex/feature-task-ui-flow` |

要求：

- 不要直接在主分支开发。
- 不要把三个人的功能混在同一个分支。
- 每个分支只做自己负责的模块。
- 如果需要改别人的接口，先沟通再改。

## 3. 代码修改边界

三名开发尽量只修改自己的目录，减少冲突。

| 开发 | 主要修改区域 | 说明 |
| --- | --- | --- |
| 开发 A | `Assets/Scripts/Gameplay/` | 分类枚举、垃圾物品、垃圾桶、投放触发、分类事件 |
| 开发 B | `Assets/Scripts/Player/`、`Assets/Scripts/Interaction/` | 键鼠移动、鼠标视角、选中、抓取、释放 |
| 开发 C | `Assets/Scripts/Core/`、`Assets/Scripts/UI/`、`Assets/Scripts/Analytics/` | 游戏状态、倒计时、HUD、反馈、结算、统计 |

不要继续把新逻辑堆进旧的：

```text
Assets/Scripts/Player.cs
Assets/Scripts/Garbage.cs
```

旧脚本可以作为参考，但 V0.3 新逻辑应拆到新目录中。

## 4. PR 规范

开发完成后提交 PR，不要自行合入。

PR 需要包含：

- 本次完成了哪些功能。
- 修改了哪些主要文件或目录。
- 如何在 Unity 中验证。
- 当前还有哪些已知问题。
- 是否影响其他开发的接口。

PR 示例说明：

```text
完成内容：
- 新增 WasteCategory、GarbageItem、TrashBin、DropZone。
- 垃圾进入桶口后能判断分类正误。

验证方式：
- 打开 MVP 场景。
- 手动拖动带 GarbageItem 的物体进入不同 DropZone。
- Console 能输出正确/错误结果。

影响范围：
- 新增 Assets/Scripts/Gameplay/ 下脚本。
- 提供 ClassificationEvents.OnClassified 给 UI 模块订阅。
```

要求：

- 不要自己 approve。
- 不要自己 merge。
- 由统筹负责人代码审核后再合入。
- 审核通过前不要继续在同一 PR 中混入无关功能。

## 5. 任务依赖关系

三个任务可以并行开发，但存在接口依赖。

### 5.1 开发 A 是接口前置

开发 A 需要优先提交一版“接口骨架”，让 B 和 C 能开始接入。

接口骨架至少包含：

```text
WasteCategory
GarbageItem
TrashBin
DropZone
ClassificationResult
ClassificationEvents
```

A 不需要一次性把所有判定细节都做完，但这些类名、字段名和事件出口要尽早稳定。

### 5.2 开发 B 依赖 A 的垃圾物品接口

B 需要使用：

```text
GarbageItem.CanInteract()
GarbageItem.SetHeld(bool held)
GarbageItem.ResetToStartPosition()
GarbageItem.isCompleted
```

B 不应该自己定义另一套垃圾分类组件，也不应该自己判断分类正误。

### 5.3 开发 C 依赖 A 的分类结果事件

C 需要订阅：

```text
ClassificationEvents.OnClassified(ClassificationResult result)
```

C 只根据分类结果更新 UI、统计和结算，不应该重新比较垃圾类别和垃圾桶类别。

### 5.4 B 和 C 之间依赖较弱

B 负责让玩家能抓取并投放垃圾。  
C 负责接收分类结果并更新 UI。

两者不需要互相等待。只要 A 的分类结果事件能触发，C 就可以工作；只要 A 的 `GarbageItem` 能被抓取，B 就可以工作。

## 6. 推荐开发节奏

### 第 1 步：A 提交接口骨架 PR

目标：

- 建好 `Assets/Scripts/Gameplay/`。
- 提供基础类和事件。
- 字段、方法名基本稳定。

这一步合入后，B 和 C 可以基于稳定接口开发。

### 第 2 步：B 和 C 并行开发

B：

- 做键鼠移动。
- 做鼠标选中。
- 做抓取和释放。
- 对接 `GarbageItem`。

C：

- 做倒计时。
- 做 HUD。
- 做正确/错误提示。
- 订阅分类结果事件。

### 第 3 步：A 补完整分类判定

目标：

- DropZone 触发分类。
- 正确后标记完成。
- 错误后允许重试。
- 分类结果完整传出。

### 第 4 步：联调

联调目标：

- B 能把垃圾送入 DropZone。
- A 能判断分类结果。
- C 能显示反馈并更新进度。

### 第 5 步：统一验收

验收目标：

- 8 个基础垃圾可完整游玩。
- 如果时间允许，补齐 12 个垃圾。
- 倒计时、投放、反馈、结算都能跑通。

## 7. 接口改动规则

如果需要修改公共接口，例如：

```text
GarbageItem
ClassificationResult
ClassificationEvents
WasteCategory
```

必须先说明：

- 为什么要改。
- 会影响谁。
- 是否有替代方案。
- 需要其他开发同步改哪些地方。

禁止：

- 私自重命名公共类。
- 私自删除公共字段或方法。
- 重复定义另一套分类枚举。
- 在自己的模块里复制一份 `GarbageItem`。

## 8. 开发控制原则

V0.3 只做键鼠可玩 MVP。

遇到新想法时按以下规则判断：

1. 不能帮助完整跑通一轮垃圾分类的功能，先不做。
2. 需要超过半天调试的视觉效果，先降级。
3. 会影响三个人共同接口的改动，先讨论。
4. 新增垃圾数量超过 12 个，先不加。
5. UI 先清楚，再美观。
6. 物理先稳定，再真实。
7. VR 适配只预留接口，不进入本阶段主线。

## 9. 合入前检查

每个 PR 合入前至少检查：

- 是否只改了自己负责的目录。
- 是否能在 Unity 中正常编译。
- 是否没有破坏其他模块接口。
- 是否有基本验证说明。
- 是否没有把临时代码、测试按钮、Debug 物体留在正式场景中。
- 是否没有新增无关功能。

最终由统筹负责人审核后合入。
